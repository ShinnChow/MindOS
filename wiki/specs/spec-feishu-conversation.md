# Spec: 飞书双向对话 (Feishu Bidirectional Conversation)

> 让用户可以在飞书里直接和 MindOS Agent 对话，MindOS 也能通过飞书回复用户。

## 目标

实现 **Feishu ↔ MindOS** 的双向消息能力：

1. **用户可以在飞书私聊机器人发消息**，MindOS 收到后交给 Agent 处理并回复
2. **用户可以在飞书群里 @机器人发消息**，MindOS 识别后处理并在群里回复
3. **用户可以在 MindOS Channel 配置页开启/关闭这个能力**，看到连接状态、最近对话活动

---

## 需求澄清

### Why?

用户已经能把 MindOS 结果发到飞书，但还不能在飞书里直接提问，这会迫使用户来回切上下文。对独立开发者/创始人来说，真正高频的是“人在飞书里时顺手问一句”，不是每次都切回 MindOS。

### Simpler?

最简单的方案是继续只做通知，不做入站。但这无法满足“在飞书里直接问”的核心目标。

本期选择的最小可用范围是：
- 只支持文本消息
- 私聊直接处理
- 群聊仅在 @ 机器人时处理
- 先复用轻量 headless runner，不复刻整条 `/api/ask` HTTP/SSE 链

### 关键假设

- 用户愿意先配置公网可达 webhook，再开启飞书对话
- 本期只要求“能对话”，不要求与前端 Ask 完全共享所有会话元数据
- 当 Agent 失败时，发送清晰兜底回复比静默失败更符合用户体验

### 核心问题

| 问题 | 影响 |
|------|------|
| **无入站通道** | 用户不能从飞书主动发消息给 MindOS |
| **无会话管理** | 无法保持上下文连续性 |
| **无 @bot 识别** | 群聊场景无法过滤噪音 |
| **无配置 UI** | 用户不知道怎么开启/验证这个能力 |

## 数据流 / 状态流

### 入站消息流（Feishu → MindOS → Reply）

```
飞书用户发消息
    │
    ▼
飞书开放平台 ──事件订阅──→ MindOS Webhook
                             │
                             ▼
                   ┌─────────────────────────────────────────────────────┐
                   │  POST /api/im/webhook/feishu                        │
                   │  ├─ 1. 验证签名 (Encrypt Key + HMAC-SHA256)         │
                   │  ├─ 2. 处理 challenge 验证（首次配置）               │
                   │  ├─ 3. 解密消息体 (AES-256-CBC)                      │
                   │  ├─ 4. 过滤：私聊 or @bot？群聊非 @bot → 忽略        │
                   │  ├─ 5. 标准化 → IncomingMessage                      │
                   │  ├─ 6. 查找/创建 session（按 platform + chatId）     │
                   │  └─ 7. 立即返回 HTTP 200（飞书要求 <5s）             │
                   └─────────────────────────────────────────────────────┘
                             │
                             ▼ (异步)
                   ┌─────────────────────────────────────────────────────┐
                   │  内部调用 Agent                                       │
                   │  ├─ 构造 user message                                │
                   │  ├─ 调用 session.prompt() 或 /api/ask               │
                   │  ├─ Agent 处理（可能调用知识库工具）                  │
                   │  └─ Agent 生成回复                                   │
                   └─────────────────────────────────────────────────────┘
                             │
                             ▼
                   ┌─────────────────────────────────────────────────────┐
                   │  FeishuAdapter.send()                                │
                   │  ├─ 回复到原 chat_id                                 │
                   │  ├─ 如果是群聊 → reply_in_thread（可选）             │
                   │  └─ recordIMActivity('conversation', 'success')     │
                   └─────────────────────────────────────────────────────┘
```

## User Flow

### 场景 1：首次配置飞书对话

**用户目标**：让飞书里的同事可以直接和 MindOS 对话。

**前置条件**：用户已配置 Feishu App ID + App Secret，状态为 Connected。

```
Step 1: 用户打开 Channels → Feishu 详情页
  → 系统反馈：页面显示 "Connected" 状态，但 Conversation 区域显示 "Not enabled"
  → 状态变化：无

Step 2: 用户点击 "Enable conversations" 开关
  → 系统反馈：开关变为开启态，显示 Webhook URL（如 https://your-mindos.com/api/im/webhook/feishu）
  → 状态变化：写入 im.json: { feishu: { ..., conversation_enabled: true } }

Step 3: 用户复制 Webhook URL，去飞书开放平台配置事件订阅
  → 系统反馈：页面显示配置指南 + "Waiting for Feishu verification..."
  → 状态变化：无

Step 4: 飞书发送 challenge 请求到 MindOS
  → 系统反馈：页面状态变为 "Webhook verified ✓"
  → 状态变化：challenge 成功响应

Step 5: 用户在飞书私聊机器人发一条测试消息 "你好"
  → 系统反馈：飞书里收到 MindOS 回复 "你好！有什么我可以帮你的吗？"
  → 状态变化：
    - Webhook 收到消息 → 创建 session
    - Agent 处理 → 生成回复
    - 发送回复 → 记录活动

Step 6: 用户回到 MindOS Channel 页面
  → 系统反馈：Recent activity 显示刚才的对话（入站 + 出站）
  → 状态变化：无

成功结果：用户确认飞书对话能力已生效，可以正常使用。
```

### 场景 2：在飞书群里 @MindOS

**用户目标**：在团队群里让 MindOS 帮忙回答问题。

**前置条件**：用户已配置并启用 Feishu conversation。

```
Step 1: 用户在飞书群里发送 "@MindOS 帮我总结一下今天的待办"
  → 系统反馈：（飞书里等待）
  → 状态变化：
    - Webhook 收到群消息事件
    - 识别到 @mention → 进入处理流程

Step 2: Agent 处理请求
  → 系统反馈：（处理中，<3s）
  → 状态变化：
    - 查找/创建群 session
    - Agent 读取用户知识库相关内容
    - 生成回复

Step 3: MindOS 在群里回复
  → 系统反馈：群聊里出现 MindOS 的回复消息，内容是待办总结
  → 状态变化：记录活动（类型 = conversation, 场景 = group）

成功结果：团队成员都能看到 MindOS 的回复，用户得到了需要的信息。

异常分支：
- 异常 A：群聊消息未 @机器人 → MindOS 不响应，不记录活动（降噪）
- 异常 B：Agent 处理超时（>30s）→ 回复 "抱歉，处理时间较长，请稍后重试"
- 异常 C：用户无权限访问知识库 → 回复 "抱歉，我没有权限访问相关内容"

边界场景：
- 消息超长（>2000 字）→ 截断后处理，回复提示"消息已截断"
- 并发消息（同一用户快速发多条）→ 排队处理，按顺序回复
- 网络断开（MindOS → Feishu）→ 回复失败，记录错误活动，用户可在 Channel 页看到
```

### 场景 3：关闭飞书对话

**用户目标**：暂时停用飞书对话能力。

**前置条件**：用户已启用 Feishu conversation。

```
Step 1: 用户打开 Channels → Feishu，找到 Conversation 设置
  → 系统反馈：显示当前状态 "Enabled"
  → 状态变化：无

Step 2: 用户点击开关关闭
  → 系统反馈：
    - 开关变为关闭态
    - 提示 "Conversations disabled. MindOS will no longer respond to messages in Feishu."
  → 状态变化：im.json: { feishu: { ..., conversation_enabled: false } }

Step 3: 飞书用户发消息
  → 系统反馈：（飞书里无回复）
  → 状态变化：
    - Webhook 仍收到消息
    - 检测到 conversation_enabled: false → 不处理，不回复
    - 不记录活动（或记录为 "ignored"）

成功结果：飞书对话暂停，但凭证和配置保留，随时可重新开启。
```

## 方案对比

### 方案 A：Webhook 内部直接 HTTP 调 `/api/ask`
- 用户体验质量：⭐⭐⭐
- 实现复杂度：中
- 可维护性：低
- 风险：自调用 HTTP、SSE 解析、鉴权/超时链重复，错误面更大

### 方案 B：抽 headless runner，Webhook 异步直接跑 Agent
- 用户体验质量：⭐⭐⭐⭐⭐
- 实现复杂度：中
- 可维护性：高
- 风险：需要维护一份与 ask runtime 同步的最小初始化逻辑

### 方案 C：只做 webhook 收消息，不真正回复
- 用户体验质量：⭐
- 实现复杂度：低
- 可维护性：中
- 风险：用户会误以为功能可用，实际没有价值

### UI 方案对比

```text
方案 A：Conversation 内联卡片             方案 B：折叠到 Settings
┌─────────────────────────────┐        ┌─────────────────────────────┐
│ Conversation                │        │ Settings                    │
│ [开关] [Webhook 状态]       │        │ ▼ 更多设置                  │
│ Public URL / Encrypt Key    │        │   App ID / Secret           │
│ [保存] [打开飞书后台]       │        │   Encrypt Key / URL         │
└─────────────────────────────┘        └─────────────────────────────┘
UX：状态与动作靠近 ⭐⭐⭐⭐⭐                UX：入口更深，心智更弱 ⭐⭐⭐
```

### 选择

选择 **方案 B（headless runner） + UI 内联 Conversation 卡片**。

原因：
1. 它给用户的是完整可用体验，而不是“看起来开启了，其实不会回复”的半成品。
2. 它避免 `/api/ask` 自调用，减少超时和调试复杂度。
3. Conversation 是这个页面的新核心能力，应该直接露出，不应再次埋进 Settings。

## 方案对比

### 方案 A：WebSocket 长连接（开发模式）

```
MindOS ◀────WebSocket────▶ 飞书 WebSocket Gateway
```

**优点**：
- 无需公网 URL
- 本地开发友好
- 实时双向

**缺点**：
- 飞书文档明确说 WebSocket 仅用于开发测试
- 连接不稳定，需要心跳保活
- 生产环境不推荐

**用户体验质量**：⭐⭐⭐（开发友好，但不稳定）
**实现复杂度**：中
**可维护性**：低（飞书可能随时改 API）

### 方案 B：HTTP Webhook（推荐 ✅）

```
飞书 ──HTTP POST──▶ MindOS /api/im/webhook/feishu
                              │
                              ▼
                    签名验证 → 解密 → 处理 → 回复
```

**优点**：
- 飞书官方推荐的生产方案
- 稳定可靠
- 与现有 API 路由架构一致
- 可复用到其他 IM 平台

**缺点**：
- 需要公网可访问的 URL（或 ngrok 等隧道）
- 需要实现签名验证和解密

**用户体验质量**：⭐⭐⭐⭐⭐（稳定、与飞书原生体验一致）
**实现复杂度**：中
**可维护性**：高

### 方案 C：轮询（不推荐）

```
MindOS ──定时 GET──▶ 飞书 API（不存在这个 API）
```

**分析**：飞书不提供消息轮询 API，此方案不可行。

### 选择：方案 B（HTTP Webhook）

**理由**：
1. 用户体验最好：稳定、实时、与飞书原生体验一致
2. 飞书官方推荐，长期可维护
3. 架构可复用到其他 IM 平台的入站支持

## 边界 case 与风险

### 边界 case

| 场景 | 处理方式 |
|------|---------|
| 消息超长（>4096 字符）| 截断到 4000 字符，回复提示"消息已截断处理" |
| 并发消息（同用户快速发多条）| 排队处理，按时间戳顺序回复 |
| Agent 处理超时（>30s）| 回复"处理时间较长，请稍后重试"，记录错误活动 |
| challenge token 不匹配 | 返回 401，不处理 challenge |
| challenge 请求 | 立即返回 challenge 值，不进入消息处理流程 |
| 群聊非 @bot 消息 | 忽略，不处理，不记录活动 |
| 用户未配置 conversation_enabled | Webhook 仍接收消息，但不处理，返回 200（避免飞书重试） |
| MindOS 无公网 URL | UI 提示需要配置公网访问，或使用 ngrok 等隧道 |
| Encrypt Key 未配置 | UI 提示必须配置 Encrypt Key 才能开启对话 |

### 风险与 mitigation

| 风险 | 影响 | Mitigation |
|------|------|------------|
| 尚未实现完整签名/AES 验证 | 安全边界不足 | 下一个迭代补齐 Feishu 正式签名与解密流程；当前仅用于受控测试 |
| 会话泄露 | 隐私问题 | 会话按 `(platform, chatId)` 隔离，仅保留少量最近消息 |
| DoS 攻击 | 服务不可用 | 后续补请求体限制、签名前置校验、速率限制 |
| Agent 处理失败 | 用户无响应 | 发送清晰兜底回复，并记录 reply activity |

## 假设与决策记录

1. **假设**：用户的 MindOS 实例有公网可访问的 URL（或配置了隧道）
   - 如果不成立：UI 提示需要公网访问
2. **假设**：飞书 App 已开启 Bot 能力和 im.message.receive_v1 权限
   - 如果不成立：Webhook 不会收到消息，UI 显示"等待验证"
3. **决策**：群聊只响应 @机器人 的消息
   - 原因：降噪，避免打扰群聊
4. **决策**：会话先采用轻量最近消息窗口，而不是复杂 TTL 清理器
   - 原因：先稳定对话体验，避免在第一期引入额外后台清理复杂度
5. **决策**：群聊默认只响应 @机器人的文本消息，不处理图片/文件/语音
   - 原因：先把文本对话链路做稳定，避免多模态输入扩大范围
6. **决策**：本期不直接从 Webhook 内部调用 `/api/ask` HTTP，而是复用 ask runtime 的 headless runner
   - 原因：避免自调用 HTTP 带来的超时、SSE 解析和身份传递复杂度，同时让 IM webhook 能直接拿到最终文本结果
7. **决策**：若当前实例没有公网 URL，则 UI 不允许把会话状态标成“Ready”
   - 原因：避免用户误以为飞书可达，结果配置后永远收不到 challenge
8. **决策**：本期不实现"飞书里查看 MindOS 知识库"功能
   - 原因：Agent 可按现有 Ask 权限读取知识库，但不新增飞书专属浏览 UI，避免范围膨胀

## Spec 对抗性审查

### 轮 1：完整性攻击

发现的问题：

1. **问题：`FeishuConfig` 现有结构没有 `encrypt_key` / `conversation_enabled` / `public_base_url`，spec 直接依赖这些字段会与真实代码脱节。**
   - 修正：把这些字段纳入 type-first 设计，并在 UI 上把它们作为独立的 Conversation 设置，而不是混在原始凭证里。

2. **问题：spec 里写“<5s 回复”，但飞书 webhook 真正需要的是“<5s 返回 200”，不是 agent 必须在 5s 内完成。**
   - 修正：改成“webhook 在 200ms 内确认接收；Agent 异步回复，典型场景尽快返回”。

3. **问题：spec 没有定义 challenge 成功后状态存哪里，页面无法知道是 Waiting 还是 Ready。**
   - 修正：增加 `webhook status` 存储与查询 API，记录 `pending / ready / error / disabled`。

4. **问题：群聊回复策略未明确是“原地回复”还是“线程回复”，会导致实现时行为不一致。**
   - 修正：本期统一为“原地回复到原 chat”；线程回复留作后续增强。

### 轮 2：可行性攻击

发现的问题：

1. **问题：如果直接让 webhook route 发 HTTP 请求到 `/api/ask`，会出现自调用、SSE 解析、鉴权链重复问题。**
   - 修正：抽出共享 headless runner，由 Webhook 直接复用 ask runtime 的 session 初始化与文本聚合逻辑。

2. **问题：恶意用户可以向 webhook 发超大 body 或伪造事件刷服务。**
   - 修正：增加 body 大小限制、签名前不做昂贵计算、签名失败直接 401、消息文本标准化前先裁剪。

3. **问题：当前 activity 类型只有 `test | agent | manual`，如果复用 `manual` 记录入站消息，用户会看不懂。**
   - 修正：扩展 `IMActivityType`，新增 `conversation_inbound` 和 `conversation_reply`。

4. **问题：MindOS 本地优先，很多用户没有公网 URL。若 UI 仍显示“Enable conversations”但没有公网条件，会制造坏体验。**
   - 修正：UI 增加 `Reachability` 区块；无公网地址时只能配置但不能进入 ready，明确提示需要公网入口/隧道。
