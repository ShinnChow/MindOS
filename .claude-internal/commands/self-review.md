对当前改动执行结构化 code review。

逐文件审查所有改动，按以下 4 个维度逐条检查，每个问题标注严重级别。

## 严重级别

- 🔴 Blocker：必须修复才能提交
- 🟡 Major：强烈建议修复
- 🟢 Minor：建议改进
- 💡 Nit：可选优化

## 检查维度

### D1 正确性

- 对照需求 / spec 逐条验收
- 查 `wiki/80-known-pitfalls.md`，确认没有重蹈已知陷阱
- 新引入的依赖版本范围是否正确？（`^` range 下界是否有需要的 API）

### D2 健壮性

- 外部调用（API / 文件 / 网络）是否有 try-catch？错误信息是否对用户有帮助？
- 用户输入是否做了验证和清洗？（空值、类型错误、注入攻击）
- 异步操作是否有超时保护？竞态条件？
- 失败路径是否有 fallback 或 graceful degradation？

### D3 架构 + 可维护性 + 精简

- **命名**：是否有 `utils`/`helpers`/`common`/`shared` 等泛化命名？→ 改为领域化命名
- **职责分离**：业务逻辑是否混在 UI 组件或 controller 里？
- **复杂度**：函数 >50 行？文件 >200 行？嵌套 >3 层？
- **Library-First**：是否有可用现成库替代的自定义实现？→ 替换
- **死代码**：未使用的 import / 变量 / 函数？→ 删除
- **重复代码**：>3 行相同逻辑？→ 提取函数
- **冗余实现**：是否有可用项目内已有函数或现成库替代的重复逻辑？→ 复用

### D4 性能 + 前端合规

- N+1 查询？大数据量 OOM 风险？
- 缓存三层覆盖？（客户端 router cache / 服务端 revalidate / 内存 cache）
- 前端改动额外检查（纯后端跳过）：
  - 设计系统合规：色值用 CSS 变量？Focus ring 用 `focus-visible:`？字体用 class？z-index 查表？
  - 状态变更三检查：条件 UI 分支互斥？state 初始值正确？disabled 守卫完整？

## 输出格式

每个问题一行：
```
[🔴/🟡/🟢/💡] 文件:行号 — 问题描述 — 修复建议
```

review 结束后输出汇总：
```
| 级别 | 数量 | 已修复 | 遗留 |
|------|------|--------|------|
| 🔴   |      |        |      |
| 🟡   |      |        |      |
| 🟢   |      |        |      |
| 💡   |      |        |      |
```

## 修复规则

- 🔴 → 立即修复 → 修复后对该文件重新检查 D1-D4
- 🟡 → 修复后说明改了什么
- 🟢 💡 → 列出但不阻塞提交
- **最多 3 轮修复循环**：第 3 轮仍有 🔴 → 停止，输出剩余问题和根因分析，交由用户决策
