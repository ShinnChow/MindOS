修复 bug。描述：$ARGUMENTS

严格按以下步骤执行，不要跳步。

## 1. 复现

- 确认能稳定复现，记录复现步骤
- 如果无法复现，先分析可能的触发条件（竞态、环境差异、数据状态）
- UI 相关 bug → 用 Playwright 截图保存当前错误状态到 `/tmp/`

## 2. 检查已知问题

- 搜索 `wiki/80-known-pitfalls.md`，看是否有同类问题和已知 workaround
- 搜索 `wiki/85-backlog.md`，看是否已在待办中
- 有已知 workaround → 评估是否该彻底修复而非继续 workaround

## 3. 写复现测试

- 先写一个能复现 bug 的测试（红灯）
- 这个测试就是修复的验收标准

## 4. 定位根因

- 不要只修表面现象，找到 **为什么** 会发生
- 画出相关的数据流 / 调用链，找到错误注入点
- 用 `software-architecture` skill 的原则审视问题区域：
  - 是否因为关注点未分离（业务逻辑混在 UI/controller 里）导致的？
  - 是否因为泛化模块（`utils`/`helpers`）职责不清导致的？
  - 如果是架构层面的问题 → 在修复时一并调整结构，不要只打补丁

## 5. 全局扫描同类问题

- 搜索代码库中同样的错误模式
- 主动排查其他文件是否存在同样的 bug
- **不要只修当前文件就结束**

## 6. 修复全部

- 修当前 bug + 扫描发现的同类问题，一次性清理
- **Two Hats 原则**（来自 `refactoring-patterns` skill）：
  - 如果修复需要先调整代码结构 → 先戴"重构帽"小步重构（每步跑测试→绿→commit）
  - 再戴"功能帽"修复 bug
  - 两件事不混着做，每步都要测试通过
- 确认复现测试变绿

## 7. 补充测试

- 除了复现测试，补充边界 case 和错误路径测试
- 过一遍 CLAUDE.md 的"边界 case 发现清单"（空值/类型/字符串/集合/时序/环境/状态）
- 重点覆盖根因分析中发现的错误注入点附近的路径

## 8. Code Review

- 执行 `/self-review`（4 维度 checklist：正确性→健壮性→架构+可维护性+精简→性能+前端合规）
- 特别关注：修复是否引入了新问题？是否影响了其他功能？

## 9. 验证

- 跑全量测试（`npx vitest run`），必须全部通过
- UI 相关 bug → 用 Playwright 截图修复后状态，与步骤 1 的截图对比

## 10. 记录 + 提交

- 将坑记入 `wiki/80-known-pitfalls.md`（模式、根因、修复方式）
- 更新 `wiki/85-backlog.md`（打勾或新增条目）
- commit + push（遵循 CLAUDE.md Git 流程，Conventional Commits，`fix:` 前缀）

## 失败处理

- **无法复现（步骤 1）**→ 加防御性代码 + 日志，记入 known-pitfalls 标记为"未确认复现"，不要盲改
- **修复引入新测试失败（步骤 9）**→ 回到步骤 6 重新修复，不要跳过 review 直接提交
- **重构步骤测试变红（步骤 6）**→ 立即 revert，尝试更小的步骤

## 最终呈现

- Bug 根因分析（数据流/调用链图）
- 修复范围（改了哪些文件、为什么）
- 全局扫描结果（同类问题是否存在、是否一并修复）
- 架构层面的改进说明（如有结构调整）
- 测试覆盖情况
- 截图对比（如有 UI bug：修复前 vs 修复后）
